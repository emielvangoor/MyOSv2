# ARM64 Learning OS — Design (Phase 0 + 1)

**Date:** 2026-06-07
**Status:** Approved

## Project goal

Build an operating system from scratch, step by step, with the primary goal of
**deep understanding** of how computers work at the lowest level. The journey
matters more than the end product. We favor transparency (seeing exactly what
the machine does) over convenience.

## Fixed decisions

| Decision      | Choice                          | Why |
|---------------|---------------------------------|-----|
| Architecture  | ARM64 (AArch64)                 | Clean modern ISA, runs in QEMU, real path to Raspberry Pi later |
| Language      | C + a little ARM64 assembly     | Transparent; nothing hidden; matches the user's C experience |
| Machine       | QEMU `virt` board               | Idealized, best-documented peripherals (PL011 UART, GIC, generic timer); no hardware quirks |
| Host          | Apple Silicon macOS + Homebrew  | Cross-compile to bare-metal ELF |
| Build system  | Hand-written Makefile           | Every compiler/linker flag is visible and explainable |
| Toolchain     | `aarch64-elf-gcc` + binutils    | Nearly all ARM bare-metal references assume it; produces ELF cleanly |

## Overall roadmap (each phase is its own spec → plan → build → understand cycle)

- **Phase 0 — Toolchain & first boot.** Install cross-compiler + QEMU; get an ELF QEMU loads and runs.
- **Phase 1 — Bare-metal "Hello, world."** Boot asm → C → UART output. *(this spec covers 0 + 1)*
- **Phase 2 — Exceptions & interrupts.** Vector table, GIC, generic timer ticks.
- **Phase 3 — Virtual memory.** Physical page allocator, then enable the MMU.
- **Phase 4 — Kernel heap.** `kmalloc`/`kfree`.
- **Phase 5 — Processes & scheduling.** Context switch, round-robin scheduler.
- **Phase 6 — User mode & syscalls.** EL0, separate address spaces, `svc`.
- **Phase 7 — Drivers & a filesystem.** virtio block device + simple FS.
- **Phase 8 — User programs & a shell.** Load/run programs; interactive shell.

We design and build one phase at a time, in order. Each produces something
visible and working, and deepens understanding before the next.

## Phase 0 + 1 detailed design

### Toolchain (Phase 0)

Installed via Homebrew, each verified before use:

- `aarch64-elf-gcc` — cross-compiler + assembler producing bare-metal ELF
- `aarch64-elf-binutils` — `ld`, `objcopy`, `objdump` (inspect what we build)
- `aarch64-elf-gdb` — source-level debugging of the kernel
- `qemu-system-aarch64` — the `virt` machine emulator

Exact installed versions will be recorded in `docs/notes/` when set up.

### Project structure

```
os/
├── Makefile              # build + run + debug commands
├── linker.ld             # memory layout: where code/data live
├── src/
│   ├── boot.S            # the first instructions that run (assembly)
│   ├── kmain.c           # kernel entry in C
│   ├── uart.c / uart.h   # PL011 UART driver
│   └── kprintf.c / .h    # minimal printf over the UART
├── build/                # generated objects + kernel.elf (gitignored)
├── docs/                 # notes + phase specs
├── .gdbinit              # debugger auto-connect config
└── compile_commands.json # editor IntelliSense (clangd)
```

### Build system — hand-written Makefile

Plain `make`. Targets:

- `make` — cross-compile and link `build/kernel.elf`
- `make run` — boot in QEMU (`-machine virt -cpu cortex-a72 -nographic`); output to terminal
- `make debug` — boot QEMU frozen and waiting (`-S -s`) for a debugger to attach
- `make gdb` — launch `aarch64-elf-gdb`, auto-connect to QEMU, break at `kmain`
- `make clean` — remove `build/`
- `make objdump` — disassemble `kernel.elf`
- `make qemu-monitor` — QEMU monitor for inspecting machine state

All compiler/linker flags (`-ffreestanding`, `-nostdlib`, `-mgeneral-regs-only`,
`-T linker.ld`, etc.) are written explicitly and explained.

### The debugging loop (centerpiece of "a good environment")

Two terminals:
1. `make debug` — QEMU starts with CPU frozen, exposing its built-in GDB stub on `localhost:1234`.
2. `make gdb` — `aarch64-elf-gdb` attaches via the stub, loads symbols from `kernel.elf`, breaks at `kmain`.

Result: breakpoints, single-stepping, `info registers`, memory examination — full
source-level debugging of bare-metal code. A committed `.gdbinit` wires this up.

### Editor IntelliSense

A `compile_commands.json` (generated by the Makefile or a small helper) so `clangd`
provides go-to-definition, autocomplete, and inline diagnostics on freestanding
kernel code. Works in VS Code, Neovim, Emacs, etc.

### What actually runs

1. **`linker.ld`** — places the kernel at `0x4008_0000` (where QEMU `virt` loads
   `-kernel` images), lays out `.text` / `.rodata` / `.data` / `.bss`, defines the stack top.
2. **`boot.S`** — entry point QEMU jumps to. Parks secondary CPU cores (only core 0
   continues), sets up the stack pointer, zeroes `.bss`, then branches to `kmain`.
   We will inspect the exception level `virt` drops us into (typically EL1) and discuss it.
3. **`kmain.c`** — calls `uart_init()`, then `kprintf("Hello, world from kernel!\n")`,
   then halts in a low-power `wfe` loop.
4. **`uart.c`** — PL011 UART driver at MMIO base `0x0900_0000` (the `virt` UART);
   writes bytes to the data register, polling the flag register for readiness.
5. **`kprintf.c`** — minimal formatted output (`%s`, `%d`, `%x`, `%c`) over the UART.

### Success criteria

- `make run` prints `Hello, world from kernel!` to the terminal.
- `make gdb` stops at a breakpoint in `kmain`; single-stepping and register reads work.
- `clangd` resolves all symbols with no false errors.

### Out of scope (deferred to later phases)

- Interrupts, timers, the GIC (Phase 2)
- The MMU / virtual memory (Phase 3) — Phase 1 runs with the MMU off
- Dynamic memory, processes, user mode, drivers, filesystem (Phases 4–8)
- Real Raspberry Pi hardware (possible later port)
